# 아키텍처 의사결정 기록 (ADR)

DRT 운영 데이터 분석 플랫폼 개발 과정에서 내린 주요 설계 결정들을 정리합니다.
각 결정은 **맥락 → 검토한 옵션 → 결정 사유 → 트레이드오프 → 결과** 순서로 기록합니다.

---

## ADR-1. 피벗 캐시 전략 — Redis TTL에서 영구 DB 캐시로 전환

### 맥락
사용자는 같은 분석 설정으로 피벗을 반복 조회하는 패턴이 많았습니다. 70+ 컬럼 × 멀티 리전 × 다차원 집계 연산이 5~30초 걸리기 때문에, 동일 요청 재계산을 줄이는 게 UX의 핵심이었습니다.

### 초기 구현
Redis 캐시 + 5분 TTL.

### 발견된 문제
- 사용자가 5분 후 같은 분석을 다시 보면 캐시 만료 → 매번 재계산
- 실제 사용 패턴은 "같은 분석을 시간 차이를 두고 여러 번 보기"였는데, 5분 TTL이 그 패턴과 맞지 않음
- 워커가 항상 바쁜 상태가 됨

### 검토한 옵션
1. **Redis TTL 연장 (예: 1시간)** — 메모리 비용 증가, 캐시 키 충돌 가능성, 만료 정책 여전히 임의적
2. **영구 DB 캐시 테이블 신설 (Pivot Data Mart)** — 저장소 비용은 들지만 명시적 invalidation 가능
3. **사용자 세션 단위 캐시** — 사용자별 격리는 좋지만 팀원 간 같은 분석 공유 시 캐시 미작동

### 결정
옵션 2 — `pivot_results` 테이블로 영구 캐시 구현. cache_key는 `{버전 prefix}:{params hash}` 구조.

### 트레이드오프
- ✅ 동일 분석 재요청 시 즉시 응답 (~95% 캐시 히트)
- ✅ 코드 버전 변경 시 prefix 교체로 일괄 invalidation 가능
- ⚠️ 저장소 사용량 증가 — 그러나 운영 데이터 분석은 분석 종류가 한정적이라 무한 증가 안 함
- ⚠️ Stale 데이터 문제 — `task_code_version` 스탬프와 누락 필드 검증 가드로 해결

### 결과
재방문 시 사용자 체감 응답 속도가 즉시 수준으로 단축. 워커 부하 감소. 다만 코드 변경 시 캐시 invalidation을 까먹으면 옛 결과가 반환되는 함정 발견 → 별도 ADR-3 참고.

## ADR-2. 멀티 리전 DB 아키텍처 — 지역 분리 vs 단일 DB의 region 컬럼

### 맥락
초기 시스템은 나주 지역만 다뤘습니다. 청주 지역 데이터가 추가되면서 "기존 시스템에 어떻게 통합할 것인가"가 결정 사항이 됐습니다.

### 핵심 제약
- 청주 데이터에는 `누적 일자`라는 추가 컬럼이 존재 (나주 스키마에 없음)
- 청주의 `날짜` 필드는 `YYYY-MM-DD HH:MM:SS` datetime 문자열 형식, 나주는 `YYYY-MM-DD` 평문 — **데이터 표현이 다름**
- 향후 추가 지역 확장 가능성 (수도권, 영남권 등)
- `dim_dates`(공휴일 캘린더)는 양 지역이 공유

### 검토한 옵션
1. **단일 DB + region 컬럼**
   - 장점: 단일 쿼리, 단순 관리
   - 단점: 스키마 차이를 한 테이블에 우겨넣어야 함 (NULL 처리, 두 형식의 날짜 혼재). 신규 지역 추가 시마다 스키마 마이그레이션 발생
2. **DB 자체 분리 (`db_naju`, `db_cheongju`)**
   - 장점: 지역별 스키마 자유. 운영 분리 (한 지역 마이그레이션이 다른 지역 영향 안 줌). 신규 지역은 새 DB 추가
   - 단점: 다중 DB 라우팅 로직 필요. 결과 통합 레이어 필요. ORM 사용 시 명시적 라우팅
3. **읽기 전용 통합 뷰 + 원본 분리** (혼합)
   - 복잡도 대비 이점 적음 → 보류

### 결정
옵션 2 — 지역별 DB 분리.

### 구현
- Django `DATABASES` 설정에 `db_naju`, `db_cheongju` 추가
- `RegionDatabaseRouter` 클래스로 지역 코드 → DB alias 매핑
- 사용자 요청 시 선택된 지역에 따라 `Model.objects.using(alias)` 호출
- `dim_dates`는 양 DB에 공통 존재, 나주 DB의 캘린더를 정본으로 사용
- 결과 통합 레이어에서 누락 컬럼은 NULL-padding (예: 나주 결과의 `누적 일자` 컬럼)
- 청주 datetime 문자열은 `__date_key__ = 날짜.str[:10]` 정규화로 일관 처리

### 트레이드오프
- ✅ 지역별 스키마 자유롭게 진화 가능
- ✅ 신규 지역 추가가 단순 (DB 등록 + 라우터 항목 추가)
- ✅ 한 지역 장애가 다른 지역 분석에 영향 없음
- ⚠️ 모든 ORM 호출 사이트에서 명시적 `using(alias)` 필요 — 누락 시 기본 DB로 잘못 라우팅되는 버그 발생 가능 (실제로 일부 view에서 발생, 회귀 테스트로 방어)

### 결과
청주 데이터 통합 후 추가로 다른 지역 확장이 발생해도 기존 코드 손대지 않고 DB 추가만으로 대응 가능. 다만 ORM 사이트별 `using(alias)` 누락 검증을 위해 정규 회귀 테스트가 필수가 됨.

## ADR-3. Mutation Testing 도입 — 커버리지의 한계 인식

### 맥락
프로젝트가 73단계까지 진행되며 테스트 코드도 함께 누적됐습니다. 코드 커버리지는 80%대를 유지했지만, **여전히 production 버그가 발생**했습니다.

### 발견된 문제
어떤 테스트는 함수 호출만 하고 결과를 검증하지 않는 형태였습니다 — 즉 코드 라인은 실행되지만 실제 동작 검증은 부재. 이런 "빈 테스트"가 커버리지에는 카운트되지만 버그를 못 잡습니다.

대표 사례: "Run Analysis 3~4번 누를 때까지 값이 달라진다"는 사용자 보고(BUG-63). 캐시 invalidation 로직에 테스트는 있었으나, 실제로 stale 워커가 만든 결과를 캐시가 받아들이는 경로는 검증 안 됨.

### 검토한 옵션
1. **테스트 코드 리뷰 강화** — 사람의 검토에 의존, 누락 가능성 지속
2. **Property-based testing (Hypothesis)** — 강력하지만 학습 비용 + 도메인 함수에 적용하기 복잡
3. **Mutation Testing (mutmut)** — 코드를 일부러 변형(`==` → `!=`, `+` → `-` 등)하고 테스트가 그 변형을 잡는지 측정. "테스트의 테스트"

### 결정
옵션 3 — mutmut 도입. 초기 kill rate **0%**에서 시작.

### 단계적 개선
1. 핵심 모듈(`tasks.py`, `views.py`, `filters.py`)에 mutation 적용
2. 살아남은 mutant 분석 → 어떤 테스트가 "보기만 하는 테스트"인지 식별
3. 해당 부분에 outcome 검증 (rendered text, return value, store mutation) 추가
4. CI 하네스에 mutation 단계 통합

### 트레이드오프
- ✅ 테스트 품질이 정량적으로 측정됨 (kill rate라는 명시적 지표)
- ✅ "이 코드는 테스트로 보호됨"이 진짜 보호되는지 알 수 있음
- ⚠️ CI 시간 증가 — 핵심 모듈만 적용으로 완화
- ⚠️ False positive (의미 없는 mutant) 처리 비용 — 무시 패턴 등록으로 관리

### 결과
**0% → 56.2% kill rate 달성.** 더 중요한 부수 효과: 테스트 작성 시 "이 코드를 변형시켰을 때 테스트가 잡을까?"를 머릿속에서 미리 시뮬레이션하는 습관이 생김. 신규 기능 개발 시 처음부터 outcome-asserting 테스트를 작성하게 됨.

---

## 종합 — 이 결정들이 보여주는 것

위 3개 결정의 공통점:
1. **초기 구현은 표준적 선택** (Redis TTL, 단일 DB, 커버리지 기반 테스트)
2. **운영하면서 한계 발견** (사용 패턴 미스매치, 스키마 차이, 빈 테스트)
3. **트레이드오프 명시적 비교 후 재선택**
4. **선택의 결과가 정량적으로 측정됨**

소프트웨어를 처음부터 완벽하게 설계할 수는 없습니다. 운영하면서 데이터를 보고, 한계가 드러나면 다시 결정하는 사이클이 자연스러운 흐름입니다. 이 프로젝트는 그 사이클을 73단계 동안 반복했습니다.
